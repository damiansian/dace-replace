### Frame 1.2: Why planning visibility matters

**Type:** Two-column

**Headline:** Accessibility should appear before the sprint plan

**Left column:**
- Heading: `If accessibility is absent`
- Scope ignores core barriers
- Debt enters the roadmap by default
- Teams discover risk too late

**Right column:**
- Heading: `If accessibility is visible`
- Risk is discussed early
- Dependencies can be planned
- Tradeoffs are explicit instead of accidental

**Speaker notes:**
Roadmap influence is not abstract strategy work. It is making sure accessibility appears when scope, budget, and timing are still adjustable.

---

### Frame 1.3: Signals to raise in planning

**Type:** Four-grid

**Headline:** What to surface during roadmap conversations

**Cards:**
- Known barriers in current patterns
- Upcoming features likely to create accessibility debt
- Dependencies on engineering or platform support
- Compliance or market risk that affects release decisions

**Key insight callout:**
Influence is easier when accessibility is framed as planning input, not late-stage objection.

**Speaker notes:**
This gives learners concrete categories to bring into planning discussions. The goal is not to say no to everything. It is to make accessibility visible soon enough to shape the decision.

---

### Frame 1.4: Adapting the message for different roles

**Type:** Three-column

**Headline:** Same case, different emphasis

**Content:**
- Executives: risk, market reach, brand, long-term leverage
- PMs: roadmap impact, prioritization, user value, delivery tradeoffs
- Engineers: implementation timing, debt reduction, rework avoidance

**Key insight callout:**
The facts stay stable. The emphasis changes based on what each audience acts on.

**Speaker notes:**
This content moves here from L21 because audience adaptation is a live-conversation skill, not a document-construction skill. In planning meetings, learners need to shift emphasis on the fly.

### Frame 2.2: What makes criteria useful

**Type:** Two-column

**Headline:** Vague intent vs testable acceptance criteria

**Left column:**
- Heading: `Weak`
- "Must be accessible"
- "Should support screen readers"
- "Needs keyboard support"

**Right column:**
- Heading: `Strong`
- Names the feature state or flow
- States expected behavior
- Can be validated in design or QA

**Speaker notes:**
Acceptance criteria matter because they turn values into implementation expectations. If the statement cannot be tested, it will not reliably survive delivery.

---

### Frame 2.3: Criteria examples

**Type:** Three-column

**Headline:** Examples of specific and testable criteria

**Content:**
- Keyboard: All interactive controls in the filter panel are reachable in logical order
- Screen reader: Error state announces problem and recovery guidance
- Visual: Focus indicator remains visible at 200 percent zoom and in high-contrast conditions

**Speaker notes:**
Use examples tied to user behavior, not vague intention. This is what helps accessibility become part of definition of done.

---

### Frame 2.4: Checkpoint - Acceptance criteria

- [ ] I understand how criteria connect accessibility to definition of done

**Speaker notes:**
This checkpoint is about precision. Learners should leave with a stronger standard for what "included" actually means.

### Frame 3.2: Early intervention vs late correction

**Type:** Two-column

**Headline:** The same issue costs more later

**Left column:**
- Heading: `Intervene early`
- Concept stage
- Pattern choice still open
- Lower rework cost
- Better team learning

**Right column:**
- Heading: `Correct late`
- Detailed design or implementation
- More sunk cost
- Higher resistance
- More compromise pressure

**Speaker notes:**
This is the core strategy lesson. Accessibility leaders need to recognize where in the lifecycle they can create the most leverage.

---

### Frame 3.3: Handling pushback

**Type:** Summary

**Headline:** When accessibility conflicts with other priorities

**Bullet list:**
- Re-anchor on user impact and known risk
- Make the debt explicit, not invisible
- Offer a concrete alternative when possible
- Clarify what will become more expensive if deferred

**Key insight callout:**
Pushback is easier to navigate when the tradeoff is named clearly instead of treated as a personal disagreement.

**Speaker notes:**
This is where advocacy becomes leadership. The goal is not to win every argument. It is to make the tradeoff explicit and defensible.
