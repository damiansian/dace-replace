Lesson 19: Contribution

Leave artifacts that outlast you

🚀 Deliver Badge 📋 Module 3H

L17 gave you the methodology for conducting reviews. L18 gave you the communication skills for delivering feedback. L19 asks a different question: what do you leave behind?

If accessibility depends on your presence in the room, it is fragile. This lesson is about making contributions that work when you are not there: reusable resources, community structures, and process improvements that raise the baseline for everyone.

💡 Key Concepts

Reusable resources 3H.01

The same accessibility issues recur because nobody contributes prevention infrastructure. Every resource you create is an issue that doesn't get filed next audit cycle.

Four types of contributions raise the baseline: checklists tied to specific component patterns or workflows, pattern guidance grounded in real team examples, onboarding materials that help new team members understand expectations, and shared examples showing paired good and bad implementations from real work.

Not all contributions are equal. Resources that get used are specific to team patterns, short enough to reference during work, grounded in the codebase, and placed where the work happens (PR templates, component docs). Resources that get ignored are generic, long, abstract, and stored in a wiki nobody visits.

✓ Checkpoint

  • I can name one reusable resource I could create for my team
  • I understand the difference between a contribution that gets used and one that gets ignored
  • I can connect the resource to a specific recurring issue pattern on my team

Communities of practice 3H.02

If accessibility depends on one person, progress is fragile. Communities of practice are the organizational structure that makes individual contributions durable.

Four elements sustain a community: shared language so the team can discuss accessibility without translation overhead, reusable examples that new reviewers can learn from, recurring critique spaces where accessibility gets discussed regularly, and onboarding pathways for new team members to build skills incrementally.

A checklist you wrote is a contribution. A monthly review session where the team uses that checklist together is a community of practice. The checklist alone can be forgotten. The recurring session keeps it alive.

✓ Checkpoint

  • I can explain the purpose of a community of practice
  • I can name at least one durable structure that spreads accessibility knowledge
  • I can connect accessibility culture to team routines, not just individual effort

Standards and systems 3H.03

Small system changes scale better than repeated explanation. Process-level contributions reach every project that uses the process, not just the ones you personally review.

Possible contributions include: accessibility acceptance criteria in user story templates, component documentation that includes accessibility requirements, PR review checklists with specific accessibility checks, and definition of done that names accessibility benchmarks.

The highest-leverage contribution is one that changes the default. If the PR template includes an accessibility checklist, every PR gets reviewed. If it doesn't, only the ones you happen to be on get reviewed.

✓ Checkpoint

  • I can identify one process or standards contribution I could make
  • I understand how process-level changes scale better than individual teaching
  • I can connect this work to reducing repeated accessibility findings

📝 Homework

Identify one concrete contribution you could make in the next two weeks. Choose from one of these tracks:

  • Resource: Create a checklist, pattern guide, or shared example tied to a recurring accessibility issue on your team
  • Community: Propose or strengthen a recurring accessibility forum, critique session, or onboarding pathway
  • Process: Draft a PR checklist addition, acceptance criteria template, or component doc update that encodes accessibility expectations
