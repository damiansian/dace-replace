### Slide 3 — Component 1: Problem Definition
**Type:** Content
**Layout:** Left text, right visual (could be a mock Jira dashboard or severity table)

```
Component 1: Problem Definition

What you're doing:
• Identify the accessibility problem your product faces
• Use real data — Jira issues, dashboard metrics, PEAS findings
• Classify findings by severity (Sev 1-4 model)
• Ground everything in data, not assumptions

Key questions to answer:
• Where is your data coming from?
• What patterns do you see across findings?
• How severe are the issues for real users?

Source lessons: L17 (review methodology, severity classification)
Learning outcomes: 3A.01, 3A.02, 3A.03
```

### Slide 4 — Component 1: What "Good" Looks Like
**Type:** Content (could mirror the checkpoint slide pattern with the callout box)

```
Problem Definition — levels of quality

Developing (2): Identifies issues, but classifies by WCAG criterion
instead of user impact. Findings are listed but not connected.

Proficient (3): Classifies by severity using Sev 1-4 model.
Connects each finding to a blocked user task.

Exemplary (4): All of the above + identifies patterns across
findings that point to a systemic root cause.

You don't need to be at a 3 or 4 right now.
You need a concrete plan and a start.
```

**Notes:** This is the rubric preview — helps them aim without grading pressure. Use the green/yellow/blue tier styling from L17's outcome tiers slide.

### Slide 5 — Component 3: Contribution Artifact
**Type:** Content
**Layout:** Left text, right examples list

```
Component 3: Contribution Artifact

Create one reusable accessibility resource for your team:

Examples:
• A component-specific annotation checklist
• An onboarding guide for new team members
• A testing protocol for your product's core workflows
• A pattern guide for your team's most-used components
• A process improvement for how bugs get triaged

It must be:
✓ Specific to YOUR team's patterns (not generic)
✓ Practical enough to use during real work
✓ Something a teammate could pick up and use

Source lesson: L19 (mentoring + culture building)
Learning outcomes: 3H.01, 3H.02, 3H.03
```

### Slide 6 — Component 3: What "Good" Looks Like
**Type:** Content (tier styling)

```
Contribution Artifact — levels of quality

Developing (2): Creates a resource, but it's generic enough to
apply to any team. Could have been pulled from the internet.

Proficient (3): Resource is clearly tailored to the team's actual
patterns, tools, and workflows. A teammate could use it today.

Exemplary (4): All of the above + artifact addresses a gap
the team didn't know it had. Demonstrates deep understanding
of how the team works.

Again — aim for a concrete plan by Monday.
Completeness comes later.
```
