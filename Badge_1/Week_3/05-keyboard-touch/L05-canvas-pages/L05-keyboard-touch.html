Lesson 05: Keyboard, Touch & Reading Order

🔍 Discover Badge 🏷️ Module 1E

Keyboard accessibility is essential for millions of users who rely on keyboard navigation, including screen reader users, voice control users, people with motor disabilities, and power users. Touch target sizing directly impacts usability for people with motor impairments, tremors, or reduced dexterity.

In this lesson, you'll learn how to ensure all functionality is keyboard accessible, design focus indicators that meet WCAG 2.2 requirements, apply proper touch target sizing, and understand how reading order affects screen reader users navigating sequentially through content.

Knowledge Objectives

  • Understand which elements must be keyboard accessible and how keyboard navigation works (Tab, Shift+Tab, Arrow keys, Enter, Space, Escape).
  • Know how SC 2.1.1 (Keyboard), SC 2.4.3 (Focus Order), and SC 2.4.7 (Focus Visible) requirements apply to design decisions.
  • Understand WCAG 2.2 SC 2.4.11 (Focus Appearance) specifications: minimum 2 CSS pixel perimeter, 3:1 contrast requirement.
  • Know WCAG 2.2 SC 2.5.8 (Target Size Minimum) requirement: 24×24 CSS pixels minimum for touch targets.
  • Recognize common keyboard accessibility failures: hover-only menus, mouse-only dropdowns, keyboard traps.
  • Understand when focus traps are intentional (modals) versus when they are failures (cannot escape component).
  • Explain that reading order is the sequence screen readers announce content sequentially, distinct from tab order which only affects interactive elements.
  • Identify common SC 1.3.2 failures: modals not hiding background content, hidden content remaining accessible, menus staying in reading order when collapsed.

  • Measure focus indicator size and contrast using browser DevTools to verify WCAG 2.2 compliance.
  • Use Stark touch target checker to identify touch targets below 24×24 pixel minimum.
  • Identify elements missing from tab order and spot keyboard traps in designs.
  • Verify logical focus order matches visual layout order.
  • Recognize when visual layouts may create reading order problems for assistive technology users.

Application Objectives

  • Audit designs for complete keyboard accessibility before handoff to engineering.
  • Design focus indicators that meet WCAG 2.2 SC 2.4.11 requirements and annotate specifications.
  • Review mobile designs for touch target compliance and document findings.
  • Create blueline annotations specifying focus indicators (color, thickness, contrast) and touch target dimensions.
  • Communicate keyboard and touch requirements to engineering teams using clear, actionable specifications.
  • Identify potential reading order issues (modals, hidden content, disabled controls) to communicate to engineering when relevant.

💡 Key Concepts

Keyboard Accessibility Fundamentals 1E.01

All interactive functionality must be operable via keyboard without timing requirements. This is a Level A requirement (SC 2.1.1) that affects millions of users including screen reader users, voice control users, people with motor disabilities, and power users.

Essential Keyboard Commands:

  • Tab: Move forward through focusable elements
  • Shift+Tab: Move backward through focusable elements
  • Enter/Space: Activate buttons and controls
  • Arrow keys: Navigate within components (menus, tabs, radio groups)
  • Escape: Close modals, cancel actions

Common Keyboard Failures: Hover-only menus, mouse-only dropdowns, custom controls missing keyboard handlers, elements removed from tab order unintentionally.

Focus Order Must Be Logical 1E.01

SC 2.4.3 requires focus order to preserve meaning and operability. Visual order should match focus order: left-to-right, top-to-bottom in most cases.

✓ Intentional Focus Traps:

Focus should be trapped within modal dialogs. This prevents keyboard users from accidentally interacting with background content.

✗ Keyboard Traps (Failures):

Users cannot escape from a component using standard keyboard commands. This is a WCAG violation.

Focus Indicators Must Be Visible 1E.02

Focus indicators show keyboard users where they are on the page. SC 2.4.7 requires visibility, and WCAG 2.2 SC 2.4.11 specifies how visible.

WCAG 2.2 SC 2.4.11 Focus Appearance Requirements:

  • Minimum perimeter: 2 CSS pixels solid or dashed
  • Contrast requirement: 3:1 against adjacent colors
  • Area requirement: As large as 1px border or 4x the border width
✓ Best Practices:

Use 3+ pixel thickness and high contrast (4.5:1 or better). Ensure visibility across light and dark modes.

✗ Common Failure:

Using :focus {outline: none} without providing a visible alternative.

Touch Targets Must Be Large Enough 1E.03

WCAG 2.2 SC 2.5.8 requires touch targets to be at least 24×24 CSS pixels. This ensures users with motor impairments can successfully tap interactive elements.

✓ Requirements ⚠️ Exceptions
  • Minimum size: 24×24 CSS pixels
  • Spacing alternative: 24 pixels between targets
  • Applies to buttons, links, form controls
  • Inline links (within paragraphs)
  • Essential controls
  • User-controlled sizing

Design Solutions: Add padding around interactive elements, increase spacing between adjacent targets, expand tap area beyond visible element using CSS.

Reading Order Fundamentals 1E.04

Reading order is the sequence in which screen readers announce content when users navigate sequentially (e.g., swiping through content on mobile or using arrow keys). This is different from tab order, which only applies to interactive elements like buttons and links.

Key insight: WCAG 1.3.2 Meaningful Sequence requires that when the order of content affects its meaning, a correct reading sequence must be programmatically determinable.

Understanding reading order vs. tab order:

  • Reading order: All content (text, images, headings, and controls) in the sequence a screen reader announces them
  • Tab order: Only interactive elements (buttons, links, form fields) in the sequence keyboard focus moves through them
Example: Disabled Controls

A disabled "Submit" button is not in the tab order, so you can't Tab to it. But it is in the reading order.

When a screen reader user swipes through the form, they'll encounter the button and hear it announced as "Submit, button, dimmed". This indicates it exists but is currently unavailable.

Common reading order failures (SC 1.3.2):

  • Modal dialogs: When a modal opens, content outside should be hidden from screen readers. If users can still read background content, that's a 1.3.2 failure.
  • Hidden content still readable: Content that is visually hidden but not properly hidden from assistive technology.
  • Menus and overlays: Dropdown content that remains in the reading order even when visually collapsed.

Designer's takeaway: You do not need to document reading order in Blueline annotations.

Reading order is primarily an engineering concern. Focus your annotations on headings, labels, and other semantic information. If a complex layout might create ambiguity, a simple note to engineering is appropriate.
