Lesson 02: Headings

🔍 Discover Badge 🏷️ Module 1B

Screen reader users cannot see visual hierarchy. They rely on programmatic heading structure to understand how content is organized, navigate efficiently, and build a mental model of the page. When headings are missing, skipped, or misused, the page becomes a wall of undifferentiated content. In this lesson, you will learn to create logical heading hierarchies and understand platform differences in heading support.

  • Understand that H1 through H6 form a hierarchy where lower numbers indicate higher importance.

💡 Key Concepts

Why Structure Matters 1B.01

Screen reader users can pull up a list of all headings on a page. They use this list to:

  • Understand page organization at a glance
  • Navigate quickly to specific sections
  • Build a mental model of the content structure

The analogy: A page without proper headings is like a book without chapter titles or a table of contents: a wall of undifferentiated content.

Heading Hierarchy Rules 1B.01

Level Purpose Example
H1 Main topic (one per page) Adobe Fonts
H2 Major sections Featured Collections, Popular Fonts
H3 Subsections within H2 Sans-Serif Fonts, Serif Fonts
H4 to H6 Deeper nesting as needed Individual font families
✓ Valid: Going down

H1 → H2 → H3 → H4

Each level nested properly

✓ Valid: Going up

H4 → H2 (closing subsection)

Can skip levels when ascending

✗ Invalid: Skipping down

H1 → H3 (skipped H2)

Breaks the outline structure

Remember: Not all large or bold text is a heading. Taglines, intro paragraphs, and promotional text are often styled prominently but are not headings.

Mobile Platform Differences 1B.02

Heading support varies by platform. Understanding these differences is critical for multi-platform design.

Platform Heading Levels User Experience
Web H1-H6 Full hierarchy, can filter by level
iOS VoiceOver H1-H6 "Heading level 2" announced
Android TalkBack Binary only Just "Heading" (no level info)

Key insight: Android TalkBack users cannot distinguish H2 from H3. What matters is whether something is a heading, not its level.

Annotation guidance for multi-platform:

  • Always specify heading levels (H1, H2, etc.) because iOS and web need them
  • Note which text should be headings for Android (the decision still matters)
  • In annotations: "H2 (heading on all platforms)"

Good news: Even on Android, marking headings helps users navigate by swiping between headings. The level just won't be announced.

Annotating Structure 1B.01

Use Stark's heading annotation feature to document structure for engineering handoff.

Include in your annotations:

  • Which text is a heading
  • What heading level it should be (H1, H2, H3, etc.)
  • Notes explaining hierarchy decisions for ambiguous cases
