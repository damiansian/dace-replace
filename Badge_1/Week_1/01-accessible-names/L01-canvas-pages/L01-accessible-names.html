Lesson 01: Accessible Names

🔍 Discover Badge 🏷️ Module 1A

Accessible names are the bridge between humans and digital interfaces. When someone uses a screen reader or voice control, the accessible name is what tells them what an element is and what it does.

In this lesson, you'll learn what accessible names are, when they're needed, how to write good ones, and how to communicate them to engineering through annotations.

📋 Learning Objectives

Learn how accessible names connect users to digital interfaces through assistive technology. This lesson covers images, icons, and interactive elements, ensuring each has a proper accessible name or is marked decorative.

Learning Outcomes

By the end of this lesson, you will be able to:

ID Outcome
1A.01 Define accessible names and explain their role in assistive technology interaction
1A.02 Identify when accessible names are required versus optional for different element types

Knowledge Objectives

  • Define what an accessible name is and explain how screen readers and voice control use them to identify buttons, links, images, and other UI elements.
  • Identify which UI elements require accessible names (e.g., buttons, links, form fields, images) and which should be marked as decorative.
  • Explain WCAG 1.1.1 (Non-text Content) and WCAG 4.1.2 (Name, Role, Value) requirements and Adobe's accessibility policies.
  • Recognize common accessible name failures such as missing alt text, unlabeled buttons, and vague link text. Describe how these create barriers for screen reader users.
  • Distinguish between functional images that need descriptive alt text and decorative images that should be hidden from assistive technology.
  • Describe the scope of accessible name barriers at Adobe and in the industry, and estimate the level of effort required to fix common issues.

Skills Objectives

  • Test designs against the Adobe Design Accessibility Checklist accessible name section with 90%+ accuracy.
  • Use Stark or equivalent tools to scan designs and identify elements missing accessible names.
  • Categorize UI elements as requiring accessible names, needing decorative treatment, or already having sufficient visible labels.
  • Spot generic link text like "Click here" or "Learn more" and flag it for content review.

Application Objectives

  • Flag accessible name issues during design reviews using the Adobe Design Accessibility Checklist.
  • Annotate designs to indicate which elements need accessible names before engineering handoff.
  • Explain accessible name requirements to team members in clear, non-technical language.

📜 Related WCAG Success Criteria

  • 1.1.1 Non-text Content (Level A): All non-text content has a text alternative
  • 4.1.2 Name, Role, Value (Level A): User interface components have accessible names
  • 2.5.3 Label in Name (Level A): Visible labels are included in accessible names

💡 Key Concepts

What is an Accessible Name? 1A.01

A short text string that identifies an element's purpose and distinguishes it from other elements on the page. Typically 1 to 3 words.

Two jobs for every accessible name:

  1. Convey purpose: What does this element do?
  2. Distinguish: Which element is this, among similar elements?

When Are Names Required? 1A.02

✓ Always ~ Sometimes ✗ Never (Mark Decorative)
  • Buttons
  • Links
  • Informative images
  • Form inputs
  • Icon-only controls
  • Navigation landmarks
  • Regions and sections
  • Groups of related content
  • Tables (when multiple)
  • Dialogs
  • Frames and iframes
  • Decorative images
  • Visual dividers
  • Background graphics
  • Icons with adjacent text
  • Redundant graphics

The core question: Does this element convey information or enable interaction? If yes, name it. If no, hide it from assistive technology.

What Makes a Good Name? 1A.03

  1. Function over form: Describe what it does, not what it looks like
  2. Concise: 1 to 3 words is ideal
  3. Distinguishing words first: Front-load unique information for scanning
  4. Consistent patterns: Same action = same verb across your interface

Specifying Names for Engineering 1A.04

Your job is to decide what each element should be called. Engineering decides how to implement it. Use Stark's Accessible Name annotation type to document your decisions clearly.

Include in your annotations:

  • The accessible name text
  • Whether the element is decorative (mark as "[n/a decorative]")
  • Notes for edge cases or implementation context

Visible Labels vs Programmatic Names 1A.05

✓ When visible text is enough:
  • Button with text label
  • Link with descriptive text
  • Form field with visible label
⚠️ When you need a programmatic name:
  • Icon-only buttons
  • Search fields without visible labels
  • Informative images
  • Ambiguous links ("Read more," "Watch now")

Label in Name rule (WCAG 2.5.3): When expanding visible text, the accessible name must start with the visible text. Voice control users say what they see.
