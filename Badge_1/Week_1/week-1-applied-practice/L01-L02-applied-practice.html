Week 1 Applied Practice: Blueline Annotations

Accessible Names + Headings Combined Project

🔍 Discover Badge 🏷️ Modules 1A + 1B 📝 28 Points

This applied practice project brings together the two foundational skills you learned in Week 1: specifying accessible names and annotating heading structure. Both skills use Stark's Blueline annotation tools to communicate accessibility requirements to engineering.

You will receive a design mockup and create a complete set of accessibility annotations covering both accessible names (for buttons, links, images, and form fields) and heading hierarchy (H1 through H6 levels). This mirrors real-world design handoff where all accessibility specifications are delivered together.

🎯 What You'll Demonstrate

Accessible Names (L01)

  • Identify which elements need accessible names vs. decorative treatment
  • Write quality names that are concise, functional, and distinguishable
  • Create clear annotations for engineering handoff
  • Know when visible text suffices vs. when programmatic names are needed

Headings (L02)

  • Design logical heading hierarchies without skipping levels
  • Annotate headings with awareness of platform differences (iOS levels vs. Android binary)

📦 Assignment Instructions

What You Receive

A design mockup in Figma containing 15 to 20 elements requiring accessibility decisions.

📎 Duplicate Figma File to Begin

The mockup includes:

  • Icon-only buttons and text buttons
  • Links with various text patterns (including "Read more" / "Learn more" links)
  • Informative and decorative images
  • Form fields with visible labels
  • Multiple content sections with headings at different levels
  • Visual hierarchy that may or may not match semantic hierarchy

What You Deliver

An annotated Figma file with Stark Blueline annotations covering:

Part 1: Accessible Name Annotations

  • All interactive elements (buttons, links, form fields) with accessible names specified
  • All informative images with accessible name specified
  • All decorative elements explicitly marked as "[n/a decorative]"
  • Expanded names for ambiguous links (e.g., "Read more about [topic]")

Part 2: Heading Annotations

  • All headings identified and marked with appropriate level (H1, H2, H3, etc.)
  • Logical hierarchy with no skipped levels going down
  • Notes for platform differences where relevant (iOS levels vs. Android binary)

Step-by-Step Process

  1. Audit the mockup: Review the entire design and identify all elements requiring accessibility annotations (interactive elements, images, text that should be headings).
  2. Annotate accessible names: Using Stark's Accessible Name annotation type, specify names for all elements. Mark decorative elements explicitly.
  3. Annotate headings: Using Stark's Heading annotation type, specify heading levels for all structural headings. Verify the hierarchy is logical.
  4. Add notes: Include notes in your annotations explaining edge cases or decisions that need context.
  5. Review: Check that all elements are covered and annotations are clear enough for engineering implementation.

💡 Tips for Success

Accessible Names

  • Function over form: "Settings" not "Gear icon"
  • Don't include the role: "Save" not "Save button"
  • Distinguish similar elements: "Edit profile" vs. "Edit preferences"
  • Mark decorative explicitly: "[n/a decorative]" with a note explaining why

Headings

  • Don't skip levels going down: H1 to H2 to H3, not H1 to H3
  • Not all big text is a heading: Taglines and intro text often aren't headings
  • One H1 per page: The main topic, usually the page title

Platform Notes

  • iOS VoiceOver: Announces "heading level 2"
  • Android TalkBack: Announces only "heading" (no level). Still specify levels for iOS/web.
