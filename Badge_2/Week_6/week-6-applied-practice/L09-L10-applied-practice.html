DOM and Accessibility Tree + Screen Reader Fundamentals Combined Project

This applied practice project brings together the two foundational skills you learned in Week 6: understanding the accessibility tree and screen reader navigation. Both skills are essential for predicting and verifying how AT users will experience your designs.

You will receive a web page example and analyze its accessibility tree, predict screen reader announcements, and identify issues that would affect AT users. This mirrors real-world practice where designers validate their work with both developer tools and AT testing.

  • Identify discrepancies between visual design and accessibility tree
  • Connect design decisions to their accessibility tree impact

Screen Reader Fundamentals (L10)

  • Predict screen reader announcements based on accessibility tree
  • Navigate content using screen reader quick navigation
  • Distinguish between browse mode and focus mode behaviors

📦 Assignment Instructions

What You Receive

A web page example with various UI components to analyze.

📎 Open Analysis Page

The page includes:

  • Interactive components (buttons, links, form fields)
  • Structural elements (headings, landmarks, lists)
  • Dynamic content (expandable sections, tabs)
  • Custom components with ARIA attributes

What You Deliver

A written analysis document covering:

Part 1: Accessibility Tree Analysis

  • Document the name, role, state, and value for 5 specified elements
  • Identify any discrepancies between visual appearance and accessibility properties
  • Explain how the accessibility tree differs from the visual DOM

Part 2: Screen Reader Predictions

  • Predict what a screen reader would announce for each element
  • Identify browse mode vs focus mode differences
  • Map out the quick navigation experience (headings, landmarks)

Part 3: Issue Identification

  • Document accessibility issues you found
  • Explain the user impact of each issue
  • Suggest design changes to address the issues

Step-by-Step Process

  1. Inspect the accessibility tree: Use browser dev tools (Chrome Accessibility panel or Firefox Accessibility Inspector) to examine each element's accessibility properties.
  2. Document properties: Record the name, role, state, and value for each specified element.
  3. Predict announcements: Based on the accessibility tree, write what a screen reader would announce.
  4. Test with screen reader: Verify your predictions using VoiceOver or NVDA.
  5. Identify issues: Document problems and suggest solutions.

💡 Tips for Success

Accessibility Tree (L09)

  • Name: What the element is called (from label, aria-label, text content)
  • Role: What type of element it is (button, link, heading, etc.)
  • State: Current condition (checked, expanded, disabled)
  • Value: Current value for inputs and controls

Screen Readers (L10)

  • Browse mode: Reading through content, all text announced
  • Focus mode: Interacting with controls, only focusable elements
  • Quick nav: Jump by heading (H), landmark (D), link (K in VoiceOver)
