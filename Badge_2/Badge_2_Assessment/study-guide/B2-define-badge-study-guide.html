Define Badge Final Assessment Study Guide

Prepare for your 25-question cumulative assessment covering all Define Badge outcomes (2A–2H).

🎯 Define Badge 📝 25 Questions ⏱️ 45 Minutes 📖 Open Book

📖 How to Use This Guide

This guide covers the key concepts, WCAG success criteria, and AT testing approaches for all eight Define Badge outcomes. Focus on understanding the why — the reasoning behind each rule — not just memorizing facts.

The assessment tests your ability to:

  • Interpret accessibility tree data and screen reader announcements
  • Identify failures across AT categories: screen readers, voice control, magnification, and WHCM
  • Apply WCAG success criteria to real-world design and testing scenarios
  • Reason through AT validation findings and prioritize remediation

Tip: Have the WCAG 2.2 Quick Reference open during your assessment: w3.org/WAI/WCAG22/quickref

🌳 Module 2A: DOM & Accessibility Tree

Reading Order vs. Focus Order

Two distinct traversal paths

  • Reading order (browse mode / virtual cursor): follows every element announced by a screen reader — headings, paragraphs, images with alt text, lists, and interactive controls. Navigated with arrow keys.
  • Focus order (Tab key): visits only interactive elements — links, buttons, and form inputs. Always produces fewer stops.
  • Tools like the Stark Blueline can visualize both paths as separate numbered sequences. More stops = reading order; fewer stops = focus order.

Screen Reader Announcements: Name → State → Role

What screen readers announce for interactive elements

  • Order: Name → State → Role (e.g., "Subscribe, dimmed, button")
  • The accessibility tree always shows the computed name — regardless of which technique provided it (visible text, aria-label, aria-labelledby, alt text)
  • No aria-label attribute ≠ no accessible name. The tree shows the resolved name.
  • VoiceOver uses "dimmed" for disabled controls. "Invalid entry" maps to aria-invalid, not disabled.

Programmatic State Must Match Visual State

Common state mismatches

  • A visually selected option (role="option") without aria-selected="true" is invisible to AT
  • aria-filled is not a real ARIA attribute — only recognized ARIA states are processed
  • aria-describedby adds supplementary description, not interactive state
  • WCAG 1.3.3 prohibits conveying state through visual characteristics (like color) alone

WCAG Success Criteria

SC Level What It Requires
1.3.3 A State not conveyed through visual characteristics alone
4.1.2 A All UI components have accessible names, roles, and states
1.3.1 A Structure and relationships programmatically determinable

Common Mistakes to Avoid

  • Confusing reading order (all content) with focus order (interactive only)
  • Assuming an element has no accessible name because there is no aria-label attribute
  • Using aria-describedby to communicate selection state instead of aria-selected
  • Leaving visual selection states without a programmatic equivalent

🔊 Module 2B: Screen Reader Navigation

Browse Mode vs. Focus Mode

Browse mode (reading / virtual cursor mode)

  • Default mode for reading web content. Navigates all content — headings, paragraphs, images, links — with arrow keys
  • Used to read static elements like headings. No mode switch required for headings.
  • NVDA/JAWS call this "browse mode"; VoiceOver uses VO+arrow keys

Focus mode (forms / application mode)

  • Engages when a user enters a form field or interactive widget
  • Keystrokes are passed directly to the element for typing. Arrow keys navigate within the widget, not the document.
  • Non-interactive content (headings, paragraphs) is not reachable by arrow keys in focus mode

VoiceOver Rotor — Structural Navigation

Using the Rotor for AT validation

  • Activated with VO+U on macOS. Presents a categorized navigation menu.
  • Categories include: Landmarks, Headings, Links, Form Controls, Tables, and more
  • Most efficient built-in tool to audit page structure without reading all content linearly
  • Not a reading mode — it is a jump-navigation overlay, separate from browse/focus modes

Know your macOS assistive technologies

Technology Purpose
VoiceOver Screen reader. Has Rotor (VO+U) for structural navigation.
Voice Control Hands-free navigation by spoken commands. "Show names" overlays accessible names on interactive elements.
Live Speech Types text and speaks it aloud — output tool, not a screen reader or navigator.
Vocal Shortcuts Custom voice-triggered actions. Not a screen reader; has no "Elements list."

Landmarks: Multiple of the Same Type

When to use unique accessible names

  • Multiple Navigation landmarks on one page are allowed — ARIA does not restrict count
  • When the same landmark role appears more than once, each instance must have a unique accessible name via aria-label or aria-labelledby
  • "Nav" shorthand in design annotations is valid — label wording doesn't determine technical compliance; unique names do

🔎 Module 2C: Screen Magnification

Reflow at High Zoom (SC 1.4.10)

Sticky elements at 400% zoom

  • At 400% zoom (= 320 CSS px viewport), stacked sticky elements consume the entire viewport, leaving no space for scrollable content
  • Fix: remove unnecessary sticky chrome — e.g., a sticky header inside a nested content panel. A global sticky header and sidebar may be justified; a third sticky header is not.
  • Adding internal scrollbars creates two-dimensional scrolling — a direct WCAG 1.4.10 failure
  • Declaring narrower support (e.g., "we only support 200% zoom") does not waive the requirement

Error Message Placement for Magnified Users

The proximity principle

  • At high magnification, users see only a small portion of the screen at once — their viewport is a narrow panning window
  • Place error messages immediately below the relevant field so both the input and its error are visible in the same magnified view
  • Grouping all errors below the Submit button forces the user to pan away from the field and back — high navigation cost
  • Pop-up windows may render off-screen, require extra dismissal steps, and are disruptive for multiple errors

Tooltip Placement for Magnified Users

Where to place tooltips in collapsed navigation

  • Place tooltips immediately adjacent to the icon (e.g., to the right in a vertical icon list) so both the icon and label are simultaneously visible in the magnified viewport
  • Tooltips placed below an icon in a vertical list risk falling outside the user's current magnified view, requiring a downward pan
  • Tooltips are valid at high magnification — the question is positioning, not whether to use them
  • Screen readers do NOT locate tooltip content by visual position; proximity benefits magnification users, not AT users

WCAG Success Criteria

SC Level What It Requires
1.4.10 AA Content reflows without loss at 320 CSS px (400% zoom) — no horizontal scrolling
3.3.1 A Input errors identified and described in text
2.4.6 AA Labels describe topic or purpose of controls

🎙️ Module 2D: Voice Control

Targeting Methods and Cognitive Load

Three targeting methods — ranked by cognitive load (low → high)

Method How It Works Cognitive Load
Name-based Speak the visible label (e.g., "Click Submit") Lowest — direct mapping, no memorization
Number overlay Invoke overlay, identify assigned number, speak it Medium — requires a mapping step
Grid overlay Speak zone number, zoom in, repeat until target reached Highest — spatial reasoning, multiple steps

Voice Control Command Grammar

Key command patterns

  • Clicking elements: "Click [visible label]" (e.g., "Click Submit")
  • Pressing keys: "Press [key name]" (e.g., "Press escape key", "Press return key", "Press tab key")
  • Browser navigation: "Go back" → triggers browser back navigation, NOT menu dismissal
  • Speaking "Escape" alone (without "Press") is not a recognized command
  • "Exit menu" and "Close menu" are not built-in Voice Control commands

Voice Control Barriers

Icon buttons without accessible names

  • Voice Control matches spoken words to accessible names in the accessibility tree
  • An icon button with no accessible name displays nothing in the Voice Control overlay — the user cannot speak a command to activate it
  • Fix: provide a programmatic accessible name via aria-label (e.g., "More options for [row item]")
  • In tables with multiple rows, each icon button needs a unique name so users can distinguish between them
  • Tooltips on hover do not help — Voice Control users cannot trigger a hover to discover a name before speaking

Label in Name (SC 2.5.3)

Placeholder text is not a persistent label

  • Placeholder text disappears the moment the user starts typing — leaving the field with no visible label
  • A persistent label: stays visible at all times, gives Voice Control users a stable name to speak, provides screen readers a reliable accessible name
  • WCAG 2.5.3 Label in Name: the accessible name must contain the visible label text — this requires a label that is actually visible
  • Tooltips on hover also fail — keyboard-only users cannot trigger hover, and tooltip content is not consistently exposed as an accessible name

WCAG Success Criteria

SC Level What It Requires
2.5.3 A Accessible name contains the visible label text
2.4.6 AA Labels describe topic or purpose
1.3.1 A Field purpose programmatically determinable

🖥️ Module 2E: Windows High Contrast Mode

How WHCM Works

Forced colors environment

  • Windows High Contrast Mode is an OS-level forced color environment — the system overrides the page's entire color palette with a limited set of system colors
  • Background fills and box-shadows are stripped. Only explicitly set border values are preserved.
  • An input field relying on background color or box-shadow for its boundary becomes invisible in WHCM dark theme — a functional failure
  • Fix: explicitly set a CSS border on input elements so forced colors maps it to a visible system color

CSS Properties That Cause WHCM Failures

Properties overridden vs. preserved by forced colors

Overridden (can cause failures) Preserved / Mapped to system colors
color, background-color border-color, outline-color
box-shadow, text-shadow, background-image Explicit border values

A selected tab that uses only background-color to indicate its state will lose that indicator entirely in WHCM — color and background-color are overridden.

WHCM System Color Semantics

WindowText vs. GrayText — the value/placeholder distinction

  • WindowText system color: applied to user-entered input values (white in dark WHCM theme)
  • GrayText system color: applied to placeholder text (appears dimmer — e.g., green in some themes)
  • The color difference between a field with a value and a field with only placeholder text is WHCM correctly communicating a semantic distinction — not an error
  • This is why it matters: WHCM applies semantic colors based on what the text is, not how you've styled it

What is and isn't a WHCM defect

  • Is a defect: Input field with no visible border in WHCM (must add explicit CSS border)
  • Not a defect: App frame borders appearing white (expected — system color mapping)
  • Not a defect: Divider lines remaining visible (structural separators should persist)
  • Not a defect: Logo rendering changing in WHCM (expected outcome of forced colors)

⚙️ Module 2F: User Preferences & Signal Chain

CSS Media Queries for OS Preferences

Know which query does what

Media Query What It Detects
forced-colors: active Windows High Contrast Mode — forced color environment. Use this to adapt WHCM.
prefers-reduced-motion User has enabled "Reduce Motion" in OS settings. Use to substitute or disable animations.
prefers-color-scheme OS light/dark theme preference. Used for dark mode — NOT for WHCM.
prefers-contrast macOS "Increase Contrast" accessibility setting. Related but NOT the WHCM query.

⚠️ prefers-reduced-color does not exist — it is not a real CSS media query.

Real User Agency: The Signal Chain

What distinguishes real agency from the appearance of it

  • Proximate: The control is reachable before the harm occurs. Auto-activating prefers-reduced-motion on page load is proximate — the user is never exposed to the triggering animation.
  • Session persistence: Preserves user data when sessions are interrupted (unrelated to motion control)
  • Pace control: Lets users slow down or pause content — different from preventing motion entirely
  • Recoverable: User can undo and return to a safe state after harm — a weaker form than prevention

Honoring prefers-reduced-motion

Correct approach when the OS preference is set

  • Replace the animation with a low-motion alternative (instant state change, subtle fade) — don't remove feedback entirely
  • Do not disable animation for all users — respect individual preference, don't impose a universal restriction
  • Do not require users to configure an inline slider at the point of interaction — the OS preference already expresses their need
  • Do not route users to a "static version" — separate-but-equal experiences violate the principle of equal access

🧪 Module 2G: AT Validation Testing

Systematic AT Validation: Programmatic AND Visual

A test can pass on one dimension and fail on the other

  • You must verify both the programmatic state (accessibility tree) and the visual presentation independently — then compare them
  • Example: A tab's aria-selected="true" may be correct in the tree, but the visual selected indicator (color fill) may fail WCAG 1.4.11 Non-text Contrast at 3:1 — the visual side fails even though the AT side passes
  • The visual affordance for a state must meet contrast requirements — not just exist — to be a valid indicator

Jira Labeling: WCAG vs. Platform Standards

Critical distinction: WHCM issues are NOT WCAG issues

  • Issues that appear specifically within Windows High Contrast Mode (e.g., missing focus states in WHCM) fall under platform/OS accessibility standards (Section 508, EN 301 549)
  • Label: Accessibility_Other_Standard — not a WCAG SC label
  • WCAG_2.4.7-Focus_Visible: focus visibility in authored styles — not the same as focus disappearing because WHCM overrides CSS
  • WCAG_1.4.11-Non-text_Contrast: contrast of UI component indicators in normal rendering — not forced-color environments

The Five Non-Interference Success Criteria

Must pass even on non-conformant pages

SC Name Severity
2.3.1 Three Flashes or Below Threshold ⚠️ MOST SEVERE — can cause epileptic seizures (direct physical harm)
2.1.2 No Keyboard Trap Blocks keyboard-only users from the rest of the page
2.2.2 Pause, Stop, Hide Auto-moving content blocks access to page content
1.4.2 Audio Control Auto-playing audio masks screen reader output
3.2.5 Change on Request Unexpected context changes without user initiation

2.3.1 is the most severe because flashing content can trigger photosensitive epileptic seizures — irreversible physical harm that no other SC can cause.

🧠 Module 2H: Perceptual & Cognitive Design

Progressive Disclosure

Adapting density and hierarchy to context

  • Progressive disclosure: present only what the user needs for their current context; reveal more complexity as needed
  • Example: desktop = full TabGroup with labeled tab bar and card grid; mobile = compact Picker + horizontal scrolling list (same content, adapted density)
  • Adjusting spatial hierarchy and information density to match viewport constraints is a core application of progressive disclosure
  • Not to be confused with focus mode (screen reader concept) or reading level (text complexity measure)

Preference Control Layers for Density

Safari Reader Mode as a model

  • Reader Mode strips navigation, ads, sidebars — re-presents only core content with clean typography and generous whitespace
  • This is a preference control layer for density and whitespace — user-controlled, runtime, adjusts visual density without changing content
  • Not a pre-ship quality gate, AI preference store, or spec annotation standard
  • The design principle: give users direct control over how much visual complexity they experience

Links Styled as Buttons — Voice Control Harm

Semantic mismatch harms Voice Control users

  • An <a> element styled visually as a button looks like a button — a Voice Control user will speak "Click [label] button"
  • The programmatic role in the accessibility tree is link, not button — so the command fails
  • Screen readers are NOT affected — they derive role from the accessibility tree, not visual appearance. Screen reader users correctly hear "link."
  • CSS classes affect visual presentation only. class="btn" does not change the semantic role exposed to AT.

💡 Test-Taking Strategies

Identify the AT Category First

  • Is the question about a screen reader, voice control, magnification, or WHCM user?
  • Each AT has different failure modes — don't apply screen reader logic to voice control questions
  • WHCM failures are platform issues, not WCAG SC failures

Watch for Plausible-but-Wrong Options

  • Options that cite real WCAG criteria but apply them to the wrong scenario
  • Options with correct conclusions but wrong reasoning (e.g., tooltip placement "because screen readers need it")
  • Non-existent ARIA attributes or CSS media queries offered as correct answers

Programmatic vs. Visual — Always Check Both

  • A correct accessibility tree does not guarantee a passing test — the visual presentation may still fail contrast
  • A visually identifiable element is not accessible if it lacks a programmatic accessible name
  • State conveyed only through color (no aria-selected) is an SC 1.3.3 failure regardless of visual clarity

Before taking the assessment, ensure you can:

  • Interpret an accessibility tree entry and predict the screen reader announcement
  • Identify the correct ARIA property for selection state vs. description
  • Distinguish browse mode from focus mode and know which is used to read headings
  • Describe what the VoiceOver Rotor is and how to use it for structural audits
  • Rank the three Voice Control targeting methods by cognitive load
  • State the correct grammar for Voice Control key press commands
  • Explain why placeholder text alone fails WCAG 2.5.3 Label in Name
  • Describe the correct fix for a WHCM input boundary failure
  • Identify which CSS media query detects WHCM — and which queries are commonly confused with it
  • Know which non-interference SC is the most severe and why (2.3.1)
