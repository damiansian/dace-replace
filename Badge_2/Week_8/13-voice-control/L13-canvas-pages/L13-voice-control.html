Lesson 13: Voice Control

🎯 Define Badge 🏷️ Module 2D

Voice control is an assistive technology used by people who can see the screen but cannot reliably use a mouse or keyboard. Unlike screen reader users who navigate sequentially, voice control users interact by speaking commands: "click Submit," "type my email," "scroll down." The core principle is simple: voice control works best when users can say what they see.

In this lesson, you'll learn how voice control tools work, identify the six design patterns that create barriers for voice users, and master the Label in Name requirement (WCAG 2.5.3), the single most important success criterion for voice control accessibility. You'll also get hands-on experience using macOS Voice Control to test designs.

📜 Related WCAG Success Criteria

  • 2.5.3 Label in Name (Level A): For user interface components with labels that include text, the accessible name contains the text that is presented visually (the key voice control criterion)
  • 4.1.2 Name, Role, Value (Level A): All user interface components must have programmatic names and roles so voice control can target them
  • 1.1.1 Non-text Content (Level A): Non-text content (icon-only buttons, images) must have text alternatives that voice control can use as targets
  • 2.4.6 Headings and Labels (Level AA): Labels must be descriptive enough for voice control users to identify and target elements
  • 2.1.1 Keyboard (Level A): All functionality must be keyboard accessible because voice control sends keyboard events to interact with elements
  • 1.3.1 Info and Relationships (Level A): Programmatic structure helps voice control tools present available interactive targets

💡 Key Concepts

How Voice Control Works

Voice control allows users to operate digital interfaces by speaking commands. Unlike screen reader users who cannot see the screen, voice control users have full vision but cannot reliably use a mouse or keyboard. They may have motor disabilities, repetitive strain injuries, temporary injuries, or situational limitations. Voice control is built into every major platform: macOS Voice Control, Windows Voice Access, Dragon NaturallySpeaking, iOS Voice Control, and Android Voice Access.

Targeting Method How It Works Speed
Name-based "Click Submit": activates the element whose accessible name matches Fast (ideal)
Number overlay "Show numbers" then "click 7": numbers appear on all interactive elements Slow (fallback)
Grid overlay "Show grid" then "click 3-7": a numbered grid covers the screen for precise positioning Very slow (last resort)

Designer Impact: Your goal is to make name-based targeting work for every interactive element. When it works, voice control is effortless. When it fails, users must fall back to slower, more cognitively demanding overlay methods.

Six Voice Control Barrier Patterns

Six design patterns account for the vast majority of voice control accessibility issues. Knowing these patterns helps you prevent barriers in your design specifications.

Barrier Example Fix
1. Missing accessible names Icon-only buttons (hamburger, close X) with no accessible name Always provide accessible names
2. Label in Name mismatch Button says "Search" but accessible name is "Find products" Accessible name must include visible text
3. Duplicate accessible names Five "Read more" links on the same page Use unique, descriptive names
4. Dynamic content Dropdown menu disappears when user speaks a command Make targets stable and persistent
5. No keyboard support Custom control only responds to mouse events Ensure keyboard operability
6. Complex interactions Drag-and-drop with no alternative Provide simple alternatives

Designer Impact: These six patterns are all preventable through design decisions. Specifying accessible names, matching labels, using unique descriptors, and providing keyboard alternatives are design responsibilities.

Label in Name (WCAG 2.5.3)

WCAG 2.5.3 Label in Name (Level A) is the core success criterion for voice control. It requires that when a UI component has visible text, the accessible name must include that text. This ensures that voice commands using the visible label will match the accessible name.

Visible Text Accessible Name Result Why
Search Search Pass Exact match
Submit Submit order Pass Starts with visible text
Close Close dialog Pass Starts with visible text
Search Find products in catalog Fail "Search" not in accessible name
Submit Send form data Fail "Submit" not in accessible name

Best Practice: The accessible name should start with the visible text. "Submit order" is better than "Order submission form: Submit" because voice users naturally say "click Submit."

Designer Impact: What you write as the visible label in your design becomes what voice users will say. If the accessible name must differ, ensure it starts with the visible text. Add accessible name annotations to your design specs.

Voice Control vs Screen Readers: Different Needs, Same Tree

Both voice control and screen readers depend on the accessibility tree, but their interaction models are fundamentally different. Understanding this helps designers write accessible names that work for both.

Screen Reader Voice Control
User can see screen? No Yes
Navigation model Sequential (reads content in order) Direct targeting (says element name)
Key requirement Descriptive accessible names Accessible names match visible text
Primary WCAG SC 4.1.2 Name, Role, Value 2.5.3 Label in Name

The Balance: Screen reader users benefit from descriptive accessible names ("Close dialog" is more helpful than "Close"). Voice control users need the accessible name to start with the visible text ("Close dialog" starts with "Close," so both needs are met). The best accessible names serve both audiences.
